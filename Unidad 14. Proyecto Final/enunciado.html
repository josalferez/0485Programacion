Aplicación de Escritorio Java
JDBC + DAO + Swing

Desarrolla una solución de gestión con arquitectura MVC, persistencia con patrón DAO y una interfaz gráfica Swing profesional sobre el dominio que elijas.
Java 17+ MySQL JDBC Swing Patrón DAO Patrón MVC 16 h de clase
🎯 Objetivo: Desarrollar una solución integral de escritorio aplicando arquitectura MVC, asegurando que la interfaz gráfica Swing esté totalmente desacoplada de la lógica de persistencia mediante el patrón DAO. El dominio del sistema es de libre elección: clínica, biblioteca, hotel, taller, tienda… siempre que cumpla los requisitos técnicos del enunciado.
Ejemplos de dominios sugeridos:
1. 🏥 Clínica Veterinaria Gestión de mascotas, propietarios, veterinarios y citas médicas. Incluye historial de visitas con diagnóstico y tratamiento.
Extensión: API de razas (TheDogAPI) para autocompletar.
2. 📚 Biblioteca Municipal Gestión de libros, socios, préstamos y reservas. Los libros tienen ISBN y ejemplares. El sistema muestra alertas de retraso.
Extensión: API de Open Library para obtener portada y sinopsis.
3. 🏋️ Gimnasio Gestión de clientes, entrenadores, clases y suscripciones. Los clientes se inscriben en clases con aforo limitado.
Extensión: IA para generar rutinas de entrenamiento según objetivo.
4. 🍕 Restaurante / Delivery Gestión de la carta, pedidos, clientes y repartidores. Los pedidos actualizan su estado en tiempo real en la tabla.
Extensión: API de geolocalización para validar dirección de entrega.
5. 🏨 Hotel Gestión de habitaciones, huéspedes y reservas. Las habitaciones tienen tipo/precio; las reservas calculan el importe total.
Extensión: API de cambio de divisas para mostrar el precio al huésped.
6. 🚗 Taller Mecánico Gestión de vehículos, clientes, mecánicos y órdenes de reparación con líneas de trabajo (mano de obra + piezas).
Extensión: IA para sugerir posibles averías según síntomas.
7. 🎬 Videoclub / Alquiler Gestión de películas, series, clientes y alquileres. Las películas tienen género, año y valoración media calculada.
Extensión: API de TMDB para obtener el póster, sinopsis y reparto.
8. 🏪 Tienda / Inventario Gestión de productos, proveedores, empleados y ventas. Alerta visual automática cuando un producto baja del stock mínimo.
Extensión: IA para predecir riesgo de rotura de stock.
9. 🌤️ Agencia de Viajes Gestión de destinos, clientes, agentes y reservas. Cálculo automático del total según número de viajeros y fechas.
Extensión: API de Open-Meteo para mostrar la previsión meteorológica.
10. ⚽ Club Deportivo Gestión de jugadores, equipos, entrenadores y partidos. La tabla de clasificación se actualiza automáticamente.
Extensión: API de Football-Data para importar ligas reales.
11. ☕ Cafetería IES Francisco Ayala Gestionar los pedidos de la cafetería del instituto, productos, usuarios y tickets de pedido.
12. 🛠️ Gestor de incidencias (Tickets) Gestionar las incidencias en equipos o instalaciones del IES Francisco Ayala, personal implicado y estado de resolución.
13. 💡 Problema del día a día Libre elección: Gestionar algún otro problema de tu día a día a través de la aplicación.

Organización del Proyecto (Estructura de Paquetes)

Es obligatorio respetar la siguiente estructura de paquetes:

src/
├── db/          ← ConexionDB.java (gestión de Connection y transacciones)
├── model/       ← Entidades del dominio (mínimo 3 clases POJO)
├── dao/         ← Interfaces + Implementaciones (un DAO por entidad)
├── dto/         ← Objetos de transferencia para consultas con JOIN
├── view/        ← Login.java · Registro.java · Principal.java
└── Main.java    ← Lanzador de la aplicación
2

Diseño de la Base de Datos

El modelo relacional debe incluir como mínimo:

›Tabla base de usuarios — username, password, email, nombre, apellidos, DNI y rol.
›Joined Table Inheritance — al menos dos tablas hijas que amplían la tabla base mediante FK primaria.
›Tabla entidad principal del dominio — cursos, productos, habitaciones, libros… con sus atributos propios.
›Tabla de relación N:M — matrículas, reservas, pedidos… con fecha y campos extra según el dominio.
›Claves foráneas + ON DELETE CASCADE — en todas las relaciones dependientes.
Ejemplo de script base (adaptar al dominio)
-- Tabla raíz de usuarios (OBLIGATORIA) CREATE TABLE usuarios ( id INT AUTO_INCREMENT PRIMARY KEY, username VARCHAR(50) NOT NULL UNIQUE, password VARCHAR(255) NOT NULL, email VARCHAR(100) NOT NULL UNIQUE, rol ENUM('cliente', 'empleado') NOT NULL DEFAULT 'cliente' ); -- Tabla hija (ejemplo) CREATE TABLE clientes ( usuario_id INT PRIMARY KEY, campo_extra VARCHAR(100), CONSTRAINT fk_cli FOREIGN KEY (usuario_id) REFERENCES usuarios(id) ON DELETE CASCADE ); -- Tabla N:M CREATE TABLE relacion ( id INT AUTO_INCREMENT PRIMARY KEY, cliente_id INT NOT NULL, entidad_id INT NOT NULL, fecha DATE NOT NULL, CONSTRAINT fk_r1 FOREIGN KEY (cliente_id) REFERENCES clientes(usuario_id) ON DELETE CASCADE );
3

Especificaciones de la Interfaz Gráfica (Swing)

A — Ventana de Login
›Componentes: campo username, JPasswordField, botón Entrar y enlace a Registro.
›Lógica: valida credenciales contra la tabla usuarios; mensaje de error si son incorrectas.
B — Ventana de Registro
›Formulario dinámico: muestra u oculta campos según el rol seleccionado en el JComboBox.
›Transacción atómica: inserta en usuarios + tabla hija en una sola transacción con rollback en caso de error.
C — Ventana Principal (Dashboard)
›JMenuBar: opciones de sesión (cambiar contraseña, cerrar sesión) y preferencias de tema.
›Panel lateral de navegación: mínimo 3 módulos de gestión (Clientes, Entidad Principal, Relación N:M, Usuarios).
›JTable central: muestra registros del módulo activo; las relaciones se resuelven con JOIN.
›Panel de operaciones adaptativo: formulario lateral que cambia campos y labels según el módulo, con botones Nuevo / Guardar / Eliminar.
›Feedback visual: JOptionPane de confirmación o error en cada operación CRUD.
4

Capa de Persistencia — Patrón DAO

›UsuarioDAO: validar(), registrar(), listarTodos(), actualizar(), eliminar()
›DAO entidad principal: insertar(), actualizar(), listarTodos(), eliminar()
›DAO relación N:M: insertar(), listarTodos() con JOIN, eliminar()
›ConexionDB: Singleton o método estático que devuelve la Connection; credenciales desde .env o .properties.
Buenas prácticas obligatorias
›try-with-resources en todos los bloques con Connection, PreparedStatement y ResultSet.
›PreparedStatement exclusivamente — prohibido concatenar SQL con valores del usuario.
›Gestión de transacciones manual — setAutoCommit / commit / rollback en operaciones de escritura múltiple.
